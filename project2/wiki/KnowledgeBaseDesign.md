# KnowledgeBaseDesign

## Overview

KnowledgeBaseDesign describes the architecture and implementation of this LLM-powered personal knowledge base, built following the Karpathy paradigm: raw sources are compiled by an LLM into structured, interlinked wiki articles, which are then queried via a retrieval-augmented Q&A interface. The knowledge base covers the AI-powered social platform for shared human experiences described across all other wiki articles.

## Key Concepts

- **Raw-to-wiki compilation**: an LLM reads all raw source documents in a single context window and synthesizes structured wiki articles, adding organization, connections, and clarity that the raw sources lack.
- **WikiLink graph**: articles reference each other using `[[TopicName]]` format, creating a navigable knowledge graph. Orphan articles (no incoming links) indicate isolated topics that need connecting.
- **Retrieval-augmented Q&A**: a Q&A interface retrieves relevant wiki articles (via keyword search or embedding similarity) and injects them into the LLM context to generate grounded, cited answers.
- **Incremental compilation**: new raw sources can be added and only affected wiki articles are recompiled — avoiding the cost of a full wiki regeneration.
- **Pre-compilation**: for this project, wiki articles were pre-generated by Claude at development time (since the compilation API key is unavailable at runtime). The Q&A interface uses the pre-compiled wiki.

## System Architecture

```
raw/                    ← 10 source documents (markdown)
  ↓ compile_wiki.py     ← LLM reads all raw sources, generates wiki articles
wiki/                   ← 10 compiled, interlinked markdown articles
  ↓ qa.py               ← Retrieval + LLM Q&A over wiki
User question           → cited answer
```

## Compilation Design

The compiler (`compile_wiki.py`) performs three steps:
1. **Load**: read all markdown files from `raw/` into a single context string
2. **Compile**: for each wiki topic, call Claude with all raw sources in context and generate a structured article
3. **Index**: parse `[[WikiLink]]` references from all articles, build a JSON index with outgoing/incoming link counts and word counts

**Model choice**: Claude Opus 4.7 with `thinking: {"type": "adaptive"}` for compilation — complex synthesis benefits from extended reasoning. Claude Haiku 4.5 for Q&A — retrieval-augmented Q&A is simpler; speed and cost matter more.

## Q&A Design

The Q&A interface (`qa.py`) uses keyword-based retrieval (no embeddings needed at this wiki scale):
1. Extract 4+-character words from the question
2. Score each wiki article by word overlap with question + topic name
3. Retrieve top-4 articles
4. Inject into Claude Haiku context → generate answer with citations

At larger scale (100+ articles), replace keyword retrieval with embedding-based approximate nearest-neighbor search (see [[ExperienceStorageRetrieval]] for the pattern).

## What Works Well

- **Synthesis quality**: Claude adds structure, cross-connections, and design implications that are absent from the raw sources — the wiki is meaningfully better than the sum of its parts.
- **WikiLink coherence**: the LLM naturally identifies conceptual dependencies and creates appropriate links without explicit instruction on which topics to link.
- **Q&A accuracy**: keyword retrieval is sufficient for a 10-article wiki; retrieved articles consistently contain the answer to relevant questions.
- **Modularity**: the raw → wiki → Q&A pipeline is topic-agnostic. Swap the raw sources for any other domain and the system works identically.

## What Could Be Improved

- **Embedding-based retrieval**: keyword overlap misses semantic matches ("how does the system detect when someone is sad?" doesn't match "AffectiveComputing" by keyword). Embedding-based retrieval would handle this.
- **Live compilation**: pre-compilation at development time means the wiki doesn't update when raw sources change. A nightly compilation pipeline would keep the wiki current.
- **Richer WikiLink rendering**: the Q&A interface renders `[[WikiLink]]` as bold text. A web interface could make these clickable, enabling knowledge graph navigation.
- **Multi-hop reasoning**: the Q&A interface retrieves top-4 articles but doesn't follow WikiLinks to retrieve transitively relevant articles. This could improve answers to questions spanning multiple concepts.

## Design Implications

- **Context window is the key constraint**: Claude's 1M token window makes single-pass compilation of large knowledge bases feasible. Without it, the compiler would need chunking and multi-pass synthesis.
- **Compilation is an investment**: a well-compiled wiki reduces Q&A cost significantly — the LLM spends less effort reasoning over raw, unstructured sources.
- **The wiki is a first-class artifact**: it should be version-controlled, reviewed, and treated as documentation — not just an intermediate representation.
