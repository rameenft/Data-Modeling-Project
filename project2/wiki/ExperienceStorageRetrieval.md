# ExperienceStorageRetrieval

The core challenge for an AI-powered social platform centered on shared human experiences is accurately representing, storing, and efficiently retrieving nuanced personal narratives. This system must bridge the gap between rich, subjective human experience and machine-readable data, enabling meaningful connections while safeguarding privacy.

## Key Concepts

*   **Dual Representation:** Storing each user experience as both a structured record (for filtering) and a dense embedding vector (for semantic similarity).
*   **LLM Extraction:** Utilizing Large Language Models to transform free-text or voice narratives into structured data and embedding vectors.
*   **Pgvector:** A PostgreSQL extension that enables efficient storage and approximate nearest-neighbor search for high-dimensional embedding vectors, crucial for semantic matching.
*   **Experience Evolution:** The recognition that personal experiences are dynamic and change over time, requiring versioning and temporal weighting in their representation.
*   **Affect Trajectory:** Tracking the emotional state associated with an experience over time, which can serve as a signal for match quality and user well-being.

## How It Works

Representing "experience" in a machine-readable form is complex. Simple tag taxonomies lose nuance, while free-text narratives are rich but hard to query effectively. The solution lies in a [[Dual Representation]] approach, combining the strengths of both:

1.  **Structured Records:** Each experience is captured in a JSON schema, allowing for precise filtering based on hard constraints. This includes fields like `category` (e.g., "health"), `subcategory` (e.g., "cancer"), `time_since_onset_months`, `duration_months`, `current_status`, and crucially, `emotional_texture` (e.g., ["fear", "isolation", "relief"]). The `emotional_texture` field, extracted by an LLM, allows for matching on the *feeling* of an experience, not just its factual category, enabling deeper [[EmpathyDesign]].

2.  **Embedding Vectors:** A dense, high-dimensional vector generated by an LLM that semantically represents the user's narrative. This captures the subtle meanings and context that structured fields might miss.

The process begins when a user shares their story, often via a [[VoiceInterface]]. Speech-to-text (STT) converts this into a transcript. An [[LLM Extraction]] pipeline then processes this narrative. It first extracts the structured JSON data, handling ambiguities like "a few years ago" by flagging for follow-up or storing nulls. Simultaneously, the LLM generates an embedding vector for the full narrative. Both the structured record and the embedding are then stored.

For storage, a PostgreSQL database extended with [[pgvector]] is used. This allows for efficient storage of dense vectors and supports approximate nearest-neighbor (ANN) search using HNSW indexes, crucial for sub-millisecond semantic similarity queries even with hundreds of thousands of experiences.
