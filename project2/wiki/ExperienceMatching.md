# ExperienceMatching

## Overview

ExperienceMatching is the core algorithmic layer of the platform, responsible for connecting users based on semantic similarity of their lived experience narratives. Unlike demographic or interest-based matching, this system treats life events — illness, grief, career disruption, immigration — as the primary matching signal.

## Key Concepts

- **Experience embedding**: a dense vector representation of a user's narrative, produced by an LLM embedding model. Two users with similar experiences have high cosine similarity in embedding space.
- **Two-stage retrieval**: Stage 1 is fast approximate nearest-neighbor search (HNSW index) returning top-100 candidates; Stage 2 is slower LLM-based re-ranking to top-5 with natural-language explanations.
- **Stable assignment**: the underlying matching problem is a two-sided market (Gale-Shapley formulation) where both mentor and mentee must accept the match.
- **Cold-start problem**: new users with no experience narrative receive matches via structured onboarding interviews that generate a synthetic initial embedding.
- **Temporal weighting**: experience records decay in relevance as time passes; the matching score weights more recent narratives higher.

## Experience Representation

Each experience is stored in two complementary forms. The **structured record** captures machine-filterable fields (category, subcategory, duration, status, outcome). The **embedding vector** captures semantic meaning, enabling matches across different phrasings of the same experience.

LLM-based extraction populates the structured record from free-text narratives (see [[ExperienceStorageRetrieval]]). Embedding computation uses the full narrative text, not the structured record — preserving emotional texture that the schema cannot capture.

## Retrieval Pipeline

```
New user narrative → embedding → HNSW ANN search → top-100 candidates
  → filter: availability, capacity, category match
  → LLM re-ranking: score top-20 on semantic fit
  → generate match explanation for top-5
  → display to user with explanation text
```

The match explanation is critical for user trust: users are more likely to accept a match when they understand why the system chose it. The explanation is generated by Claude, grounding the recommendation in specific shared experience elements.

## Design Implications

- **Explanation-first design**: never show a match without a human-readable explanation. Opaque recommendations undermine trust in sensitive contexts.
- **Opt-in granularity**: users should control which experiences are visible for matching — some experiences may be too sensitive to share with strangers.
- **Feedback loop**: every accepted or rejected match is a training signal. Poor matches should trigger experience extraction review.
- **Mentor capacity enforcement**: matching must respect mentor load limits; no mentor should receive more active mentees than their configured capacity.
- **Privacy by default**: candidate lists are anonymized until mutual match acceptance; raw narratives are never transmitted to other users.
